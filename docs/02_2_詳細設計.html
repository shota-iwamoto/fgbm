FORT-GBM v2 C++・GPU拡張 詳細設計書

Financial Ordered Robust Student-t Gradient Boosting Machine

C++20による計算コア、CPU並列実行、CUDA/HIPを中心とするGPUバックエンド、 複数GPU分散、Python binding、永続化、数値再現性および障害時fallbackを定義する。

GPU実装はCPU版と同一の数学的目的関数を評価する。 ただし浮動小数点reduce順序、近似histogram、特殊関数実装、 era圧縮モードによってbit単位の一致は保証されない。 決定論モードでは固定reduce方式を使用し、許容誤差内の数値一致を保証対象とする。

1. 設計目標

1.1 機能目標

  • 回帰、分類、ranking、quantileを単一のC++コアで実行する。
  • Student-t MM objectiveを標準実装する。
  • capped era-level KL-DROをCPU・GPU双方で実行可能にする。
  • era別histogramとrobust splitをCPU・GPUへ実装する。
  • factor penaltyとturnover penaltyを疎演算として実装する。
  • target maturityを遵守するordered categorical encodingを提供する。
  • 完全目的関数line searchによって候補木を受理または棄却する。
  • 単一GPU、複数GPU、CPU fallbackを同一APIから選択できるようにする。

1.2 非機能目標

  • C++20、RAII、例外安全、明示的所有権を基本とする。
  • CPU/GPU間の不要な同期と転送を避ける。
  • GPUメモリ不足時に安全な分割実行またはCPU fallbackを行う。
  • 決定論モードと高速非決定論モードを分離する。
  • 公開ABIを薄いC ABIで安定化し、内部C++ ABIへ依存させない。
  • モデル形式をCPU/GPUに依存させない。

3.1 設計原則

  • 数学ロジックとexecution backendを分離する。
  • CPUとGPUで同一の設定、モデル、診断形式を使用する。
  • GPU kernelの直接呼出しを上位training層へ露出しない。
  • backend固有型を公開APIへ持ち込まない。
  • device同期はbackend API内に閉じ込める。

7.2 所有権規則

  • Bufferはallocationの共有所有権を持つ。
  • kernel起動中のbuffer解放を防ぐため、event依存をmemory poolが管理する。
  • 非所有viewはstd::spanまたはDeviceSpanを使う。
  • host pointerをdevice kernelへ暗黙に渡さない。
  • 非同期転送元host memoryはpinned allocationまたは同期完了まで生存させる。

12.1 Runtime polymorphism

Objective選択は学習開始時の一回だけなのでvirtual dispatchを許容する。 サンプル単位の計算はvirtual関数内部からtemplate化されたCPU loopまたはGPU kernelへ渡し、 行ごとのvirtual dispatchは行わない。

13. Student-t CPU kernel

// File: src/objectives/student_t_cpu.cpp
#include <algorithm>
#include <cmath>

namespace fortgbm {

void ComputeStudentTDerivativesCpu(
    const double* target,
    const double* prediction,
    const EraIndex* era,
    const double* scale_by_era,
    std::size_t row_count,
    double degrees_of_freedom,
    double curvature_floor,
    double curvature_ceiling,
    double* gradient,
    double* curvature
) {
    const double nu = degrees_of_freedom;

    #pragma omp parallel for schedule(static)
    for (std::int64_t index = 0;
         index < static_cast<std::int64_t>(row_count);
         ++index) {
        const std::size_t i = static_cast<std::size_t>(index);
        const double residual = target[i] - prediction[i];
        const double sigma = scale_by_era[era[i]];
        const double denominator =
            nu * sigma * sigma + residual * residual;

        gradient[i] =
            -((nu + 1.0) * residual) / denominator;

        const double mm_curvature =
            (nu + 1.0) / denominator;

        curvature[i] = std::clamp(
            mm_curvature,
            curvature_floor,
            curvature_ceiling
        );
    }
}

}

13.1 SIMD

era別scaleのgatherが必要なため、AVX2/AVX-512 gather命令または era連続row layoutを利用する。eraごとに連続配置されている場合、 scaleをregisterへ保持して内側loopをvectorizeする。

14. Student-t CUDA kernel

// File: gpu/cuda/objective_kernels.cu
#include <cuda_runtime.h>

namespace fortgbm::cuda {

__global__ void StudentTDerivativeKernel(
    const double* target,
    const double* prediction,
    const EraIndex* era_by_row,
    const double* scale_by_era,
    std::size_t row_count,
    double degrees_of_freedom,
    double curvature_floor,
    double curvature_ceiling,
    double* gradient,
    double* curvature
) {
    const std::size_t index =
        static_cast<std::size_t>(blockIdx.x) * blockDim.x
        + threadIdx.x;

    if (index >= row_count) {
        return;
    }

    const double residual =
        target[index] - prediction[index];

    const double sigma =
        scale_by_era[era_by_row[index]];

    const double denominator =
        fma(
            degrees_of_freedom,
            sigma * sigma,
            residual * residual
        );

    gradient[index] =
        -((degrees_of_freedom + 1.0) * residual)
        / denominator;

    const double raw_curvature =
        (degrees_of_freedom + 1.0)
        / denominator;

    curvature[index] =
        fmin(
            curvature_ceiling,
            fmax(curvature_floor, raw_curvature)
        );
}

}

15. Era Loss Aggregation

15.1 数式

\[ L_e = \sum_{i:e_i=e}a_i\ell_i. \]

15.2 CPU実装

eraごとにrowが連続しているため、era単位のparallel loopで競合を避ける。

15.3 GPU実装

各eraを一つ以上のthread blockへ割り当てるsegmented reductionを使用する。 大規模eraは複数blockのpartial sumを生成し、二段階reduceする。 

Kernel A:
    one or more blocks per era
    compute weighted sample loss
    write partial era sums

Kernel B:
    reduce partial sums for each era
    write final era loss

15.4 Deterministic reduction

  • strict modeでは固定tree reductionを使用する。
  • atomicAddによる順序不定reduceは使用しない。
  • fast modeではdouble atomicまたはcooperative reductionを許可する。

16.1 解法

  1. loss降順のcap付き線形最大化解を計算する。
  2. そのKLが半径内ならKL制約inactiveとして返す。
  3. それ以外は温度 \(\tau\) を外側二分探索する。
  4. 各 \(\tau\) で正規化dual \(\eta\) を内側二分探索する。
  5. 総和、KL、capのKKT残差を確認する。

16.2 GPU実行方針

era数が小さい場合、GPU kernel起動コストが支配的になる。 次の規則でbackendを選択する。

条件 実行場所
era数が閾値未満 CPUでsolver、era lossだけ非同期転送
era数が閾値以上 GPUで全solver
複数GPU global era lossをall-reduce後、rank 0または全GPUで解く

16.3 GPU kernel分割

LinearMaxKernel
    compute cap-constrained extreme distribution

NormalizationKernel
    evaluate q_e(tau, eta)
    output partial sum

KlKernel
    evaluate q_e * log(q_e / pi_e)
    output partial KL

RiskKernel
    evaluate q_e * loss_e

Host or device controller
    update eta and tau bounds

CUDA Graphを使う場合、固定回数のbisectionをgraph化する。 厳密な早期終了はmaskで無効化し、graph topologyを固定する。

方式 用途

21.3 Era tile

一度に扱うera数を \(E_t\) とし、shared memory量を、

\[ M_{\mathrm{shared}} = B_t \left[ S_{\mathrm{aggregate}} + E_tS_{\mathrm{era}} \right]

とする。device shared memory上限から \(B_t,E_t\) をautotuneする。

21.4 Atomic precision

  • mixed modeでもglobal histogramはdoubleを標準とする。
  • double atomic未対応deviceはCAS実装またはpartial reduceを使用する。
  • strict modeではglobal atomic順序依存を避ける。

21.5 Histogram subtraction

親histogramから小さい子histogramを引き、大きい子を生成する。 GPUではvectorized subtraction kernelを使用する。

\[ \operatorname{Hist}(R) = \operatorname{Hist}(P) - \operatorname{Hist}(L). \]

負の微小曲率は許容誤差内なら0へclipする。 超過時は大きい子histogramを再構築する。

24. GPU Robust Split評価

24.1 Era改善vector

\[ \begin{aligned} \Delta_e={}& G_{P,e}v_P+\frac12H_{P,e}v_P^2 \\ &- G_{L,e}v_L-\frac12H_{L,e}v_L^2 \\ &- G_{R,e}v_R-\frac12H_{R,e}v_R^2. \end{aligned} \end{div>

24.2 二段階評価

全candidateへ完全DRO solverを実行すると高コストになるため、 数学的に安全なscreeningと完全評価を分離する。

  1. 曲率、sample数、coverageによる不適格候補を除外する。
  2. 安価な下界またはsoft-min近似で候補を絞る。
  3. 上位 \(K\) 候補に対してcapped KL-DROを完全に解く。
  4. 最終候補を元era粒度とdouble precisionで再評価する。
screeningで真の最良候補を除外しないため、 使用するscreening値は上限・下限関係が証明できるもの、 または明示的な近似モードに限定する。 標準のresearch_exactでは全有効候補を完全評価する。

24.3 Candidate parallelism

  • era数が小さい場合：一candidateを一warp。
  • era数が中程度：一candidateを一block。
  • era数が大きい場合：candidate-era二次元grid。

24.4 Weighted MAD

47. Property-Based Test

  • DRO半径0で \(q=\pi\) になる。
  • DRO radius増加に対してrobust riskが非減少になる。
  • era capを厳しくすると最大敵対重みが非増加になる。
  • coverage prior増加に対して葉値絶対値が非増加になる。
  • turnover penaltyが0ならturnover勾配が合成されない。
  • factor直交予測ではfactor penaltyが数値誤差内で0になる。
  • 受理された固定objective木で完全目的関数が非増加になる。

50. C++・GPU受入基準

  1. C++20のCPUコアで全主要objectiveが実行できること。
  2. Student-t MM、DRO、factor、turnoverの数学テストを通過すること。
  3. CUDA backendでobjective、histogram、split、predictionが実行できること。
  4. GPU era lossとDRO weightがCPU参照値に許容誤差内で一致すること。
  5. GPU robust splitがCPUと同じbest splitを選択すること。
  6. 同値score時のtie-breakがCPU/GPUで一致すること。
  7. 全受理木について完全objectiveの非増加条件を満たすこと。
  8. GPU OOM時に設定どおり縮退または安全に失敗すること。
  9. 単一GPUモデルをCPUとGPUの双方で推論できること。
  10. 保存形式がGPU architectureに依存しないこと。
  11. reproducible modeで同一環境の再学習結果が一致すること。
  12. 複数GPUでera lossとhistogramのcollective結果が単一GPU参照値と一致すること。

51. 最終構成

\[ \boxed{ \begin{aligned} \mathrm{FORT\mbox{-}GBM\ v2} ={}& \mathrm{C{++}20\ Core} + \mathrm{CPU\ SIMD/Parallel\ Backend} \\ &+ \mathrm{CUDA/HIP\ Execution\ Backend} + \mathrm{Device\ Resident\ Training} \\ &+ \mathrm{Student\mbox{-}t\ MM} + \mathrm{Capped\ Era\ KL\mbox{-}DRO} \\ &+ \mathrm{Era\mbox{-}Tiled\ Histogram} + \mathrm{GPU\ Robust\ Split} \\ &+ \mathrm{Factor/Turnover\ Sparse\ Kernels} + \mathrm{Exact\ GPU\ Line\ Search} \\ &+ \mathrm{Multi\mbox{-}GPU\ Collective\ Training} + \mathrm{Point\mbox{-}in\mbox{-}Time\ Safety}. \end{aligned} } \]
C++実装では数学層、木学習層、execution backendを分離する。 GPUは単なる標準GBDT histogram高速化ではなく、era別統計、 capped KL-DRO、coverage prior、factor penalty、turnover penalty、 完全目的関数line searchまで含めてdevice上へ拡張する。 動的な木構造制御は初期段階ではCPUに置き、 大規模なサンプル演算、histogram、split候補評価、予測およびreduceをGPUへ委譲する。 これにより数学的仕様を維持しながら、単一GPU、複数GPU、 out-of-coreおよびCPU fallbackへ段階的に対応する。
